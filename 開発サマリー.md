# 開発サマリー

このファイルは、「日米Amazonアービトラージツール」のこれまでの開発内容と今後のタスクをまとめたものです。後から作業を再開する際のガイドとしてお使いください。

## 1. 概要

- **目的:** 日本のAmazonの商品を米国のAmazonで無在庫販売するためのWebアプリケーション。
- **技術構成:**
  - **フロントエンド:** React, TypeScript, Material-UI
  - **バックエンド:** Node.js, Express
- **プロジェクトルート:** `amazon-arbitrage-tool/`

## 2. 起動方法

**注意:** フロントエンドとバックエンドは、それぞれ別のターミナルで起動する必要があります。

### バックエンド (APIサーバー)

1. ターミナルで `amazon-arbitrage-tool/backend` ディレクトリに移動します。
2. `npm start` コマンドを実行します。
3. `Server is running on http://localhost:3001` と表示され、プロセスが待機状態になれば成功です。

### フロントエンド (画面)

1. ターミナルで `amazon-arbitrage-tool/frontend` ディレクトリに移動します。
2. `npm start` コマンドを実行します。
3. 自動的にブラウザで `http://localhost:3000` が開きます。

## 3. 設定ファイル

### APIキーと認証情報

- **場所:** `amazon-arbitrage-tool/backend/.env`
- **内容:** Amazon SP-APIの各種認証キーを設定します。

### アプリケーション設定

- **場所:** `amazon-arbitrage-tool/backend/config/settings.json`
- **内容:** 送料、手数料、為替レート、在庫監視の閾値、除外リスト、**リードタイムバッファ(n値)** などの設定が保存されます。

## 4. 本番デプロイ手順

本番環境（Render等）へデプロイする際の重要事項です。

### データベース (PostgreSQL) の準備
- 本番環境ではデータの永続性を確保するため、必ず **PostgreSQL** を使用してください。
- クラウド環境のファイルシステムは再起動でリセットされるため、JSONファイルベースの保存は使用禁止です。
- 環境変数 `DATABASE_URL` を設定することで、自動的に PostgreSQL モードで起動します。

### 環境変数の設定 (Environment Variables)
デプロイ先の管理画面で以下の変数を設定してください。

**バックエンド (Server):**
- `NODE_ENV`: `production`
- `DATABASE_URL`: PostgreSQLの接続文字列
- `FRONTEND_URL`: フロントエンドのURL (例: `https://your-app.vercel.app`)
- `JWT_SECRET`: 任意の長い文字列 (認証用)
- `SELLING_PARTNER_APP_CLIENT_ID` / `_SECRET`: Amazon SP-API認証情報
- `SPAPI_AWS_ACCESS_KEY_ID` / `_SECRET_ACCESS_KEY`: AWS IAM認証情報
- `SPAPI_ROLE_ARN`: IAMロールARN
- `KEEPA_API_KEY`: Keepa APIキー

**フロントエンド (Client):**
- `REACT_APP_API_URL`: バックエンドのAPIエンドポイントURL (例: `https://your-backend.onrender.com/api`)

### インスタンス数の制限
- **重要:** バックエンドのインスタンス数は必ず **「1」** に固定してください。
- 複数インスタンスで起動すると、Cronジョブ（価格調整・在庫監視）が二重に走り、Amazon APIのスロットリングやデータの整合性エラーの原因となります。

### CORSとリダイレクトの設定
- `FRONTEND_URL` を正しく設定することで、ブラウザからの通信（CORS）とAmazon連携後の戻り先URLが自動的に構成されます。

### アカウント（ユーザー）管理について
- フランチャイズ（SaaS）展開にあたり、セキュリティと課金管理の観点からフロントエンドの「新規登録画面」は削除されています。
- お客様のアカウントを作成・確認する場合は、サーバー（ローカルまたは本番環境）のコンソールで以下のコマンドを実行します。
  ```bash
  # backend ディレクトリにて実行
  node scripts/manage-users.js
  ```
- 画面の指示に従いユーザーを追加し、発行された「ユーザー名とパスワード」をお客様へお伝えください。

## 2026/05/07 開発・調査サマリー

### 1. SP-API検索の重複ループ修正

- `research_log_log_1777565614111.csv` を確認したところ、1000件中ユニークASINが20件のみで、同じ20件が繰り返し取得されていた。
- 原因はCatalog Items APIのページング指定で、`nextToken` を送っていたためSP-API側で無視され、1ページ目が繰り返し返っていたこと。
- `backend/src/services/sp-api-client.js` を修正し、`pageToken` を使うように変更。
- 取得済みASINとページトークンの重複検知も追加し、同じページの無限ループを防止。

### 2. 利益設定エラーの意味

- 「販売価格に対応する利益設定が見つかりません。」は、計算された販売価格が `profitabilityTiers` のどの価格帯にも入らない場合に出る除外理由。
- 例: 利益ルールが `0円〜10000円` だけの場合、販売価格が10000円を超える商品は利益計算対象外になる。
- 対策として、想定販売価格の上限まで価格帯別利益ルールを設定する必要がある。

### 3. 現在出品中商品の確認

対象:

- ASIN: `B009L14CGM`
- SKU: `AUTO-1777562056166-B009L14CGM`

確認結果:

- ローカル管理上は価格 `42.15 USD`、数量 `6`。
- Amazon Listings API上でも `DISCOVERABLE` / `BUYABLE`、価格 `42.15 USD`、数量 `6` を確認。
- productTypeは `FOOD_STORAGE_CONTAINER`。
- 既存ASINを再出品しようとした場合、単品出品では既存SKUを再利用する。
- 一括出品側に重複ガードが不足していたため、`/bulk-listing-from-asins` に既存ASINチェックを追加し、同一ASINが管理中ならスキップするように修正。

### 4. 価格・在庫自動更新の修正

- `price-adjuster.js` で、価格が変わらない場合に数量だけが変わってもAmazon側へ反映されない問題を修正。
- 価格または数量のどちらかが変わった場合に出品オファーを更新するようにした。
- 価格更新時にリードタイムがデフォルト値へ戻らないよう、計算済みリードタイムも `putListingsItem` に渡すように修正。

### 5. 為替レートの自動取得

- 手入力だった為替レートを外部APIから取得できるようにした。
- 追加ファイル: `backend/src/services/exchange-rate-service.js`
- 設定項目を追加:
  - `autoExchangeRateEnabled`
  - `exchangeRateRefreshIntervalMinutes`
  - `exchangeRateUpdatedAt`
  - `exchangeRateDate`
  - `exchangeRateSource`
- API追加:
  - `POST /api/settings/exchange-rate/refresh`
- フロント設定画面に以下を追加:
  - 為替自動更新ON/OFF
  - 更新間隔
  - 今すぐ取得ボタン
  - 最終取得日時表示
- 自動取得がONの場合、手入力欄は無効化。

### 6. 利益下限による自動取下げ・再出品

- 利益基準を下回った出品を自動でAmazonから削除し、`suspended_listings.json` に退避する仕組みを追加。
- 利益基準を再び満たした場合、次回の価格調整サイクルで再出品する処理を追加。
- 対象ファイル:
  - `backend/src/services/listing-manager.js`
  - `backend/src/services/price-adjuster.js`
  - `backend/src/services/inventory-watcher.js`
- 在庫閾値による取下げは現時点では自動再出品対象外。利益基準による取下げのみ再出品対象。

### 7. 国際送料DDP・FSC対応

- `ddp.xlsx` を確認し、US DDPシートの送料表を設定へ反映。
- FSCは `0.204`、固定加算は `220円` として初期設定。
- `shippingCostTiers` はDDP表のFSC前の基本運賃として登録。
- 送料計算は以下に変更:
  - 基本運賃 + 基本運賃 × FSC + 固定加算
- フロント設定画面に以下を追加:
  - FSC率
  - 固定加算額
  - 重量別送料テーブル
- 既存設定に送料テーブルがない場合、自動でデフォルト送料テーブルを補完するマイグレーションを追加。

### 8. 重量パース不具合の修正

- 1kgを超える商品でも送料が2370円になる問題を確認。
- 原因は重量文字列 `kilograms` を `kg` として認識できず、`1.495 kilograms` を `1.495g` と誤判定していたこと。
- `profit-calculator.js` の重量パースを修正し、以下に対応:
  - `kg`, `kilogram`, `kilograms`
  - `g`, `gram`, `grams`
  - `ounce`, `ounces`, `oz`
  - `pound`, `pounds`, `lb`, `lbs`
- 検証例:
  - `0.5 kilograms` は500g枠
  - `1 kilograms` は1000g枠
  - `1.495 kilograms` は1500g枠

### 9. リサーチログ取得漏れ対策

- `research_log_log_1778078991438.csv` を確認。
- 992件中ユニークASINは992件で、以前のページングループは解消済み。
- ただし後半で価格・重量・体積などの未取得が多かった。
- SP-APIの価格取得・属性取得にリトライ処理とチャンク間待機を追加。
- 429、5xx、ネットワーク系エラーは最大3回リトライする。
- 属性未取得時にCSVへ `undefined` が出ないよう、デフォルト値を設定。

### 10. リサーチ中のUIリフレッシュ対策

- リサーチ中にブラウザ更新するとUI上から実行中ジョブが消える問題を修正。
- `DashboardPage.tsx` で検索キューを `localStorage` に保存するようにした。
- ブラウザ更新後も実行中/待機中ジョブが画面に残る。
- 完了ログが作成されたら、対応するジョブを自動でキューから削除する。
- 注意: ブラウザ更新後は `AbortController` は復元できないため、画面表示の維持が主目的。

### 11. リサーチログ画面の更新ボタン・削除修正

- リサーチログ画面の更新ボタンが点滅する問題を修正。
- 原因は `onLogsLoaded` の参照が毎回変わり、ログ取得effectが再実行され続けていたこと。
- `DashboardPage.tsx` で `useCallback` を使い、callback参照を固定。
- 更新ボタンは不要と判断し、画面から削除。
- ログは初回表示時と削除後に自動取得する仕様に変更。
- フロントには削除ボタンがあったが、バックエンドに削除APIがなかったため追加。
- 追加API:
  - `DELETE /api/research-logs/:logId`
- `ResearchLogsTab.tsx` の文字化け表示も正常な日本語へ修正。

### 12. APIキー・認証情報のGit管理対策

- コード内にAPIキーやシークレット値の直書きは見当たらなかった。
- APIキー類は `process.env` から読み込む実装。
- 確認した主な環境変数:
  - `KEEPA_API_KEY`
  - `AMAZON_CLIENT_SECRET`
  - `SELLING_PARTNER_APP_CLIENT_SECRET`
  - `SPAPI_AWS_ACCESS_KEY_ID`
  - `SPAPI_AWS_SECRET_ACCESS_KEY`
  - `JWT_SECRET`
- `backend/config/user_.../amazon_auth.json` にAmazonのrefresh token / access tokenが保存されていることを確認。
- `.gitignore` に以下を追加:
  - `backend/.env`
  - `backend/config/`
  - `*.csv`
  - `*.xlsx`
- すでにGit管理されていた `backend/config` 配下は、実ファイルを残したまま `git rm --cached -r backend/config` で追跡解除。
- 今後の運用方針:
  - APIキー類は `.env`
  - ユーザー設定・ログ・Amazon認証トークンは `backend/config/`
  - どちらもGitへコミットしない
- もし過去にGitHubなど外部へpush済みの場合は、Amazon SP-API/LWAの認証トークンを再発行・失効することを推奨。

## 2026/04/28 本日の修正・進捗まとめ

### 1. 出品エラーの解消とカテゴリー指定の最適化
- **ProductTypeの修正:** `getCatalogItemsByAsins` でカテゴリー名がオブジェクトとして取得されていた問題を修正。
- **出品の安定化:** 特定のカテゴリー（例: `TOWEL`）を指定するとAmazonから詳細な商品情報を求められるため、汎用的な `PRODUCT` に固定して送信するように変更。これにより、在庫更新がエラーで止まるリスクを回避。
- **オファー更新の明示:** 既存ASINへの相乗り出品であることを示す `LISTING_OFFER_ONLY` 属性を正しく設定。

### 2. 在庫数（数量）の動的反映
- **日本Amazonの在庫追従:** 日本の出品者数（`jpSellerCount`）を取得し、これを出品時の在庫数（`quantity`）として設定するロジックを実装。
- **安全装置:** 日本の在庫が0や不明な場合でも、Amazon側で出品が無効化されないよう、出品時には最低でも「1」の在庫を設定して送信。

### 3. リードタイム（お届け日）の自動計算
- **設定追加:** フロントエンド「設定」タブに「リードタイムバッファ (n値)」を追加。
- **自動計算:** `日本の出荷作業日数 + n値` を計算し、米国のリードタイムとして設定。デフォルトは日本2日 + バッファ3日 = 計5日。

### 4. 自動管理機能（価格・在庫）の強化
- **価格調整機能 (`price-adjuster.js`):** 15分おきの価格更新時に、価格だけでなく日本の最新在庫数も再取得してAmazonに反映させるように強化。
- **在庫監視機能 (`inventory-watcher.js`):** 1時間おきに在庫切れ・赤字転落をチェックし、基準外の商品は自動で出品を取り下げる（削除する）ことを確認。
